# 전파정책 전문가 AI — 차별화 아이디어 요청

> SK텔레콤 기술정책팀 | 2026-05-27  
> **이 문서의 목적**: 현재 시스템의 현황과 한계를 솔직하게 공유하고,  
> "타 부서 챗봇·일반 GPT와 확실히 다른, 팀 업무 생산성을 실제로 높이는 툴"로 발전시키기 위한 아이디어를 구합니다.

## 1. 팀장이 제기한 핵심 질문

> *"이걸 만들어서 내가 왜 써야 하지?  
> 어차피 일반 GPT에 물어봐도 답변해 주고, 정확한 현행법은 법제처 가서 더블 체크하면 되잖아."*  
> — 2026-05-27 팀장 대화록 中

이 질문이 이 문서의 출발점입니다.  
현재 시스템이 무엇을 하고 있는지, 어디서 막히는지, 그리고 **어떤 방향에서 진짜 차별점을 만들 수 있을지**에 대한 외부 시각이 필요합니다.

## 2. 현재 시스템이 하는 일

### 2-1. AI 자문 (핵심 기능)

85개 공개 문서(전파법 계열 78개 + 전기통신사업법 4개 + 정보통신망법 3개)를 지식 베이스로 구축하여, 조문 원문 근거를 동반한 전파·통신 정책 자문을 제공합니다.

```
사용자 질문
    ↓
법령·고시 RAG 검색 (업로드된 원문에서 직접 인용)
    ↓
Claude API (claude-sonnet-4-6) 답변 생성
    ↓
근거 조문 번호 + 내용 + 해석 + 한계 명시
```

**설계 원칙:**
- 조문 추정·생성 금지 — 업로드된 원문에서 확인된 내용만 인용
- 확인 불가 시 `"law.go.kr 직접 확인 필요"` 명시
- 할루시네이션 방지를 위해 범용 AI와 달리 '검증된 DB 기반' 답변만 출력

### 2-2. 모닝 브리핑 (자동화 기능)

매일 오전 18개 키워드로 웹 검색을 수행하여 전파·통신 분야 뉴스를 수집, 요약하여 Supabase DB에 저장합니다.

```
18개 키워드 검색 (IT전문지 + 경제지 + 정부사이트)
    ↓
최근 7일 이내 기사 필터링 + 중복 제거
    ↓
카테고리 분류 (정책/기술/표준/법령)
    ↓
신규 기술 용어 감지 → tech_terms DB 저장
    ↓
브리핑 텍스트 생성 (주요뉴스 + 주목포인트 + 새 용어)
    ↓
daily_briefings 테이블 저장
```

### 2-3. 기술 용어 사전

뉴스·자문 과정에서 등장하는 전파·통신 전문 용어를 자동 감지하여 DB에 누적합니다.  
현재 약 23개 용어 등록 (5G SA, NTN, HAPS, IMT-2030, O-RAN, AI-RAN 등).

## 3. 팀장이 지적한 현재의 문제점 (솔직한 자기 진단)

### 문제 1 — "이게 일반 AI와 뭐가 다른가?"

| 비교 항목 | 일반 AI (Claude/Gemini) | 현재 이 시스템 |
|-----------|------------------------|---------------|
| 답변 속도 | 매우 빠름 | **느림** (RAG 검색 단계 多) |
| 법령 최신성 | 불확실 (학습 컷오프 의존) | **검증된 문서 기반** (단, 수동 업데이트) |
| 조문 정확도 | 높지만 보장 없음 | **원문 인용, 추정 금지** |
| 접근성 | 누구나 즉시 사용 | 별도 웹 대시보드 |
| 업무 맥락 반영 | 없음 (범용) | **기술정책팀 용어·법령 특화** (예정) |

**팀장의 핵심 지적:** "느리고, 쓸 수 있는 상황이 제한적이면, 굳이 이 툴을 열 이유가 없다."

### 문제 2 — "데일리 브리핑이 정보 피로도를 오히려 높인다"

팀장은 이미 매일 아침 네이버 뉴스, 카카오톡 공유, 사내 공지 등으로 정보 과부하를 경험하고 있습니다. 센터 내 타 팀의 AI 발표에서도 "뉴스 보여주기" 류가 대부분이었다고 언급했습니다.

**현재 브리핑의 한계:**
- 뉴스를 재가공해서 보여줄 뿐, "그래서 우리 팀이 오늘 뭘 해야 하는지"는 말하지 않음
- 중요하지 않은 날도 동일한 형식으로 브리핑이 생성됨
- 팀 업무 액션(보고서 업데이트, 검토 의견 제출, 내부 공유 등)과 연결되지 않음

### 문제 3 — "법령 자동 갱신이 안 된다"

법령·고시가 개정되면 **담당자가 직접 PDF를 다운로드하여 Claude Projects에 재업로드해야** 합니다. 모니터링 브리핑으로 개정 소식은 포착하지만, 지식 베이스 자체가 자동으로 갱신되지는 않습니다.

> *"법이 바뀐다는 얘기가 나오면 모니터링에 뜰 거고, 관리자가 보고 검증해서 지식 베이스에 넣어야 한다는 거네."* — 팀장

### 문제 4 — 팀 내 확산이 안 됨

현재는 단일 사용자 구조(개발자 본인 사용)입니다. 팀원 로그인, 권한 관리, 사용 이력 공유 기능이 없어 팀 전체 툴로 확장하기 어렵습니다.

## 4. 현재 구현된 차별점 (강점)

외부 AI 도구와 비교했을 때 현재 시스템이 실제로 갖고 있는 강점:

**① 조문 원문 인용 보장**  
일반 AI는 법령을 요약·재구성하지만, 이 시스템은 업로드된 원문에서 직접 인용합니다. 조문 번호뿐 아니라 그 조문이 무엇을 규정하는지, 담당자 업무와 어떻게 연결되는지를 함께 제시합니다.

**② 전파·통신 특화 지식 베이스**  
85개 문서를 기술정책팀 실무 관점에서 선별했습니다. 타 부처 내부 행정 문서(68개)를 의도적으로 제외하고, 주파수·전자파·적합성평가·ITU-R 등 핵심 도메인에 집중했습니다.

**③ 인식론적 구분**  
"확정된 사실 / 해석 / 추정 / 정책 의견"을 구분하여 제시하고, 법적 결론은 단정하지 않습니다. 보고서에 직접 붙여 쓸 수 있도록 "검토 초안" 수준의 출력을 설계했습니다.

**④ 전파정책 특화 시스템 프롬프트**  
핵심 조항 혼동 사례(주파수 재할당 이용기간, 폐업 고지 기한 등)를 명시적으로 구분하는 규칙이 내장되어 있습니다.

## 5. 팀장이 제시한 차별화 방향 3가지

회의에서 나온 아이디어로, 아직 구현되지 않은 영역입니다.

### 아이디어 A — "뉴스 때문에 오늘 우리 팀이 뭘 해야 하는지" (액션 아이템 연동)

단순 브리핑이 아니라, 포착된 이슈가 우리 팀 업무에 미치는 영향을 분석하고 행동 지침을 제안합니다.

```
예시:
[뉴스] 과기정통부, 1.8GHz 주파수 재할당 조건 발표
    ↓
[현재 브리핑] "재할당 대가 14.8% 인하, 5G SA 의무화 확정"
    ↓
[목표 브리핑] "재할당 대가 14.8% 인하, 5G SA 의무화 확정
               → 기술정책팀 오늘 할 일:
               ① 재할당 고시 원문 확인 (전파법 시행령 제43조 대조)
               ② 사내 SA 전환 일정 검토 보고서 업데이트 필요
               ③ 1.8GHz 3년 단기할당 의미 → 2029년 재검토 시점 캘린더 등록"
```

### 아이디어 B — 복수 법령 연쇄 효과 분석 (교차 검증)

단일 법령 Q&A가 아니라, 하나의 실무 상황에서 여러 법령이 어떻게 교차 적용되는지를 한 번에 보여줍니다.

```
예시 질문: "주파수 재할당 거부 통보를 받았을 때 절차는?"
    ↓
[현재] 전파법 제17조(재할당) 설명
[목표] 전파법 제17조(재할당) + 제11조(취소·회수) + 행정심판 가능 여부
       + 과거 유사 사례 히스토리 + 시효·기한 요약 → 한 화면에 정리
```

### 아이디어 C — 크리티컬 이슈만 즉시 알림, 일반 뉴스는 생략 (알림 최적화)

정보 피로도를 낮추는 핵심은 "이슈가 없는 날엔 브리핑을 보내지 않는 것"입니다.

```
현재: 매일 자동 브리핑 발송 (이슈 유무 무관)
목표: 
  - 크리티컬 이슈(주파수 할당 고시 변경, 전파법 시행령 개정 등) → 즉시 알림 (텔레그램/메일)
  - 일반 뉴스 → 주 1회 요약으로 묶기
  - 이슈 없는 날 → 브리핑 생략, 조용히
```

## 6. 아이디어를 구하는 구체적인 질문

이 문서를 읽는 분께 다음 질문에 대한 의견을 구합니다.

**Q1. 차별화 방향의 우선순위**  
A·B·C 세 방향 중 "이 팀에서 가장 먼저 체감 효과를 낼 수 있는 것"은 무엇이라고 생각하시나요? 그 이유는?

**Q2. 팀장이 지적한 "진짜 가려운 곳"**  
팀장은 "출장 기차표 예매 타이밍 알림"을 예로 들며 "업무에서 진짜 귀찮은 걸 긁어줘야 한다"고 했습니다. 전파정책 담당자의 일상 업무에서 AI가 긁어줄 수 있는 "진짜 귀찮은 것"이 무엇일지 아이디어가 있으신가요?

**Q3. 법령 자동 갱신 파이프라인**  
법령·고시 개정을 감지하고 지식 베이스를 자동으로 갱신하되, 할루시네이션 위험 없이 신뢰도를 유지하는 현실적인 방법이 있을까요?

**Q4. "느린 속도" 문제**  
RAG 검색 + 법령 원문 인용 구조를 유지하면서 응답 속도를 개선할 수 있는 방법이 있을까요? 아니면 속도보다 정확도를 강조하는 UX 전략이 더 현실적일까요?

**Q5. 팀 전체 확산 전략**  
현재는 개발자 1인 사용 구조입니다. "팀 전체가 매일 쓰는 툴"로 만들기 위해 가장 중요한 조건이 무엇이라고 생각하시나요?
